## #001 — BigQuery 日本語カラム名のSQL実行エラー

| 項目 | 内容 |
|------|------|
| **日付** | 2026-03-18 |
| **コミット** | f34b179 |
| **重要度** | 高（データ取得が完全に失敗する） |

### 事象
BigQueryに接続済みの状態で「売上教えて」と質問すると、「データ取得ツールに問題が発生」というエラーが表示され、データが取得できない。

### 原因
GeminiがBigQueryのSQLを生成する際、日本語カラム名（`月`、`合計`、`A事業部門` など）をバッククォート（`` ` ``）で囲まずに出力するケースがあった。

- 例: `SELECT SUM(合計) FROM ...` → BigQueryが `Illegal input character "\345"` エラーを返す
- システムプロンプトでバッククォートの使用を指示していたが、`SUM()` 等の関数内のカラム名には適用されないことがあった
- エラーメッセージ「データ取得ツールに問題が発生」はコード内に定義されたものではなく、SQLエラーを受けたGeminiが生成したテキスト

### 解決策
SQL実行前に日本語識別子を自動的にバッククォートで囲む処理を追加。

1. **`domain/sql_validator.py`** に `auto_backtick_japanese()` 関数を新設
   - SQL中の日本語を含むトークンを正規表現で検出し、`` ` `` で囲む
   - 文字列リテラル（`'...'` / `"..."`）やバッククォート内はスキップ
   - `validate_sql()` の最終ステップとして自動適用
2. **`orchestration/reasoning_engine.py`** でTool Call実行前に `validate_sql()` を呼ぶよう変更
   - バリデーションエラー時はGeminiにエラーを返して再試行させる

### 影響範囲
- BigQueryに日本語カラム名を持つテーブルが存在するすべての企業データ
- `SELECT *` のみのクエリは影響なし（カラム名を直接指定しないため）
- `SUM()`, `WHERE`, `GROUP BY` 等で日本語カラム名を使うクエリが影響を受けていた

### 修正ファイル
| ファイル | 変更内容 |
|---------|---------|
| `domain/sql_validator.py` | `auto_backtick_japanese()` 追加、`validate_sql()` に統合 |
| `orchestration/reasoning_engine.py` | SQL実行前に `validate_sql()` を呼び出し、エラー時のハンドリング追加 |

### 検証結果
| テストケース | 修正前 | 修正後 |
|-------------|--------|--------|
| `SELECT SUM(合計) FROM ...` | Illegal input character エラー | `SUM(\`合計\`)` に自動変換 → 成功 |
| `SELECT 月, A事業部門 FROM ...` | Illegal input character エラー | `\`月\`, \`A事業部門\`` に自動変換 → 成功 |
| `SELECT * FROM ...` | 成功（影響なし） | 成功（変化なし） |
| `WHERE 月 = '2025年1月'` | — | `\`月\`` に変換、文字列リテラル内はスキップ → 成功 |
| 既にバッククォート済み | 成功 | 二重付与なし → 成功 |

### 再発防止
- `validate_sql()` をSQL実行の唯一の入口として統一したため、今後のSQLもすべて自動修正対象になる
- Geminiのプロンプト指示だけに頼らず、コード側で確実にカバーする防御的設計に変更

## #002 — Geminiがツールを呼ばずにテキスト回答を返す（不安定動作）

| 項目 | 内容 |
|------|------|
| **日付** | 2026-03-18 |
| **コミット** | 未コミット（#001と同セッション） |
| **重要度** | 最高（同じ操作で結果が変わり、信頼性を損なう） |

### 事象
BigQuery接続済みで「売上教えて」と質問すると、データを取得できる場合と「データ取得ツールに問題が発生」とエラーになる場合がランダムに発生。同じ質問・同じデータなのに結果が異なり、デモや実運用で信頼できない状態。

### 原因
Gemini APIのFunction Calling（ツール呼び出し）は**確率的な判断**で行われる。システムプロンプトで「必ず`query_bigquery`を呼べ」と指示していても、Geminiがその指示を無視してテキストだけで回答するケースがあった。`temperature=0.0`でもツール呼び出しの判断には揺れがある。ログ上、全セッション中**7回**ツールが呼ばれなかったケースを確認。

### 解決策
Gemini APIの`ToolConfig`で`mode="ANY"`を指定し、**API仕様レベルでツール呼び出しを強制**した。

- **初回呼び出し**: `mode="ANY"` — Geminiは必ずFunction Callを返す（テキスト回答不可）
- **2回目以降**（データ取得後）: `mode="AUTO"` — テキスト回答やさらなるツール呼び出しを自由に判断

```python
# 初回: ツール強制
tool_config_forced = types.ToolConfig(
    function_calling_config=types.FunctionCallingConfig(mode="ANY")
)
# 2回目以降: 自動判断
tool_config_auto = types.ToolConfig(
    function_calling_config=types.FunctionCallingConfig(mode="AUTO")
)
```

### 影響範囲
- BigQuery接続時のすべての質問応答
- BQ未接続時は`tools=None`のため影響なし

### 修正ファイル
| ファイル | 変更内容 |
|---------|---------|
| `orchestration/reasoning_engine.py` | `ToolConfig(mode="ANY")`で初回ツール強制、データ取得後に`AUTO`へ切替 |

### 検証結果
| テストケース | 修正前 | 修正後 |
|-------------|--------|--------|
| 「売上教えて」×複数回 | 成功したり失敗したり（7/全セッション中で失敗） | `mode="ANY"`によりAPI仕様レベルで100%ツール発火を保証 |

### 再発防止
- プロンプト指示（自然言語）ではなく、APIパラメータ（`ToolConfig.mode`）で動作を制御
- `mode="ANY"`はGemini APIの仕様として確定的にツールを呼ぶため、確率的な揺れは発生しない
- #001（バッククォート問題）と合わせ、LLMの不確実性はすべてコード側で吸収する方針を確立

## #003 — mode="ANY" + ストリーミングによる応答遅延・SQL無限繰り返し

| 項目 | 内容 |
|------|------|
| **日付** | 2026-03-18 |
| **コミット** | 未コミット（#002の後続修正） |
| **重要度** | 最高（応答が3分以上返らない、同じSQLが無限実行される） |

### 事象
`mode="ANY"`でツール強制を導入後、以下の2つの問題が発生：
1. **応答遅延**: Gemini APIの`generate_content_stream` + `mode="ANY"`の組み合わせで、応答が3分以上返らないケースが頻発
2. **SQL無限繰り返し**: ストリーム中に同じfunction_callが複数チャンクで返され、同一SQLが5〜8回繰り返し実行される

### 原因
`mode="ANY"`と`generate_content_stream`（ストリーミング）の組み合わせが不安定。
- ストリーミングでは応答がチャンク単位で返るため、function_callの境界が曖昧になる
- 同じfunction_callが複数チャンクに分散して返り、全て実行してしまう
- whileループ構造と相まって、ツール結果を返した後もGeminiが再度function_callを返し、無限ループに陥る
- `mode="ANY"`自体が悪いのではなく、**ストリーミングとの組み合わせ**が根本原因

### 解決策（v3.1 — ゼロから再設計）

**Call-1を非ストリーミングに変更し、Call-2のみストリーミングを使用する2フェーズ固定構造。**

```
Call-1: generate_content（非ストリーミング）+ mode="ANY"
  → 1回のAPI呼び出しで完結した応答を受け取る
  → function_callが1つだけ確実に含まれる
  → SQL実行は1回のみ

Call-2: generate_content_stream（ストリーミング）+ tools=None
  → ツール結果を含む履歴を渡してテキスト回答を生成
  → ストリーミングでリアルタイム表示
```

| 項目 | 旧（v1〜v2） | 新（v3.1） |
|------|-------------|-----------|
| Call-1 | ストリーミング | **非ストリーミング** |
| mode | ANY（ストリームと相性悪い） | ANY（非ストリームなら安定） |
| Call-2 | ストリーミング | ストリーミング（同じ） |
| ループ | while True | **なし（if/else固定）** |
| SQL実行回数 | 不定（繰り返しあり） | **1回固定** |
| Gemini呼び出し回数 | 不定 | **最大2回固定** |

### 影響範囲
- BigQuery接続時のすべての質問応答
- BQ未接続時はtools=Noneのため影響なし

### 修正ファイル
| ファイル | 変更内容 |
|---------|---------|
| `orchestration/reasoning_engine.py` | `stream_events`をゼロから再設計。Call-1を非ストリーミング化、whileループ廃止、2分岐固定構造に変更 |

### 再発防止 — 今後の修正で必ず守るルール
1. **`generate_content_stream` + `mode="ANY"` の組み合わせは絶対に使わない**
2. **`mode="ANY"`を使う場合は必ず`generate_content`（非ストリーミング）で使う**
3. **whileループでGemini APIを呼び出さない** — ループは構造的に無限実行のリスクがある
4. **Gemini呼び出し回数は固定値で設計する** — 「最大N回」ではなく「必ずN回」
5. **ストリーミングはテキスト回答生成（最後のCall）にのみ使用する**
