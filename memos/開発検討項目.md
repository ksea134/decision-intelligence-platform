### 検討001: 本番運用向けの定量評価・品質管理基盤

**背景**: 顧客に本番機として渡す際、応答時間の定量評価と回答プロセスの品質管理が必要。

**現状の課題**:

| 領域 | 現状 | 問題 |
|------|------|------|
| 応答時間 | UIにスピナーで表示するだけ | 記録・蓄積されない。統計分析不可 |
| 回答品質 | 評価なし | 正しさ・出典の適切さ・ユーザー満足度が不明 |
| エラー監視 | logger出力のみ（レベル不備で未出力） | エラー率・空応答率が追跡できない |
| コスト | 記録なし | トークン使用量・API呼び出し回数が不明 |
| 監査証跡 | なし | 誰が・いつ・何を質問したかのログがない |

**検討すべき対応案**:

1. **最低限（本番投入前に必須）**
   - loggerレベル修正（INFO以上を出力）
   - 応答時間・エラー率のファイルorBQへの記録
   - 回答プロセス各ステップの成功/失敗追跡

2. **中期（Phase 7相当）**
   - Cloud Monitoring統合（レイテンシ、エラー率ダッシュボード）
   - トークン使用量・APIコスト記録（FinOps）
   - ユーザーフィードバック収集（Good/Badボタン）

3. **長期**
   - Vertex AI Evaluation（回答品質の自動評価）
   - Groundedness評価（出典と回答の整合性チェック）
   - Human in the Loop（専門家レビューフロー）

**ステータス**: 設計完了 → 詳細は `memos/層別仕様_AI Ops＆Observability層.md` セクション8〜11に記載

### 検討002: BigQueryツールのタイムアウト仕様（2026-03-20確定）

**背景**: チャット入力で業務と関係ない質問（例:「明日の天気は？」）を送信した際、ADKエンジンのgeneral_agentがquery_bigqueryツールを呼び出し、全テーブルSELECT*を実行するため90秒以上かかる。

**決定事項**:
- `query_bigquery`ツールに30秒のタイムアウトを設定する（B案）
- タイムアウト発生時は取得済みデータのみで回答を生成する
- 開発ルールC07（LLMにツール実行判断を委ねない）に準拠し、ツール側で制御する

**ユーザー向け仕様（PoCで伝える必要あり）**:
- BigQueryからのデータ取得は最大30秒で打ち切られる
- データ量が多い場合、一部のテーブルのみ取得して回答する場合がある
- 業務と無関係な質問でも応答は返るが、データ取得の待ち時間（最大30秒）は発生する

**ステータス**: 確定・実装済み

### 検討003: 出典情報（Grounding）の正確性改善（2026-03-20）

**背景**: 出典情報が「AIが実際に参照したデータ」ではなく「AIに渡された全データ」になっている。全テーブルSELECT*で全データをプロンプトに一括注入するアーキテクチャでは、「参照した」と「利用可能だった」の区別がつかない。

**現状の問題**:
- 使っていないデータが出典に表示される（全テーブル補完）
- 使っているのに出典に出ないことがある（LLMの書き忘れ）
- Gemini Grounding機能はプロンプト直接注入のデータには効果なし（対象外）

**パターン別正確性**:
| パターン | 正確性 | 理由 |
|---------|--------|------|
| スマートカード+データソースフィルタ | 比較的高い | 不要データが渡されない |
| 全データ注入（チャット入力） | 低い | 全データが見えるためノイズが多い |

**対策ロードマップ**:
| 対策 | 内容 | 時期 |
|------|------|------|
| 米印方式に戻す | LLMに「使った箇所に※を付けろ」と指示。BQ全テーブル補完を削除 | 今（実装中） |
| データソースフィルタの拡大 | チャット入力でも質問内容に応じてデータを絞る | 中期 |
| Retrieval粒度の改善 | 質問に応じたテーブル選択・GCS関連ファイル選択 | 中期 |

**誤った認識の訂正**:
- Gemini Grounding機能 → DIPのプロンプト直接注入方式には効果なし
- 「RAG方式に移行」→ DIPは既にRAG。正しくは「Retrievalの粒度を細かくする」

**ステータス**: 米印方式は実装済み。Retrieval粒度改善は検討006で詳細設計。

---

**方針決定事項**:
- Google Cloud標準機能を徹底活用（Cloud Logging / Cloud Trace / Cloud Monitoring / Cloud Alerting）
- DIP外部からの独立監視（DIP障害時にも監視基盤は影響を受けない）
- アラート通知はお客様IT部門 + Kyndryl運用チームの両方に発報
- 実装タイミングはPhase A〜D（本番投入前〜中長期）の段階的実装

### 検討004: Looker統合（タイル内ミニチャート＋詳細ビュー埋込）

**背景**: DI Dashboardのスマートカードに、Lookerのデータを動的に表示し、クリック時にLookerダッシュボードの詳細ビューに遷移する機能。

**設計方針（Claude.aiで検討済み）**:
- Layer 1: Looker APIでデータのみ取得 → DIP側（Chart.js）で描画（タイル内ミニチャート）
- Layer 2: タイルクリック → Looker Embed SDKで詳細ダッシュボードをDIP内パネルに展開
- iframe埋め込みは却下（タイルサイズで崩れる、デザイン統一不可）

**実装時の注意点**:
- Rechartsではなく**既存のChart.js**で統一すること（ライブラリ混在回避）
- Looker API認証はサーバーサイド（FastAPIプロキシ）で行い、フロント側に秘密情報を渡さない
- 企業ID・権限をLookerユーザー属性に渡し、行レベルセキュリティを適用

**前提条件（実装着手前に確認必須）**:
- お客様がLooker契約を持っているか
- BigQuery直接接続+Chart.jsで代替できないか検討してから判断
- スマートカード（AI質問ショートカット）とLookerタイル（データ表示）の役割整理

**事業フェーズ**: 3（本番運用）。PoC段階では不要。お客様がLookerを使っている場合に初めて価値がある。

**ステータス**: 検討中（未着手）。詳細仕様は元ドキュメント削除済み、要点はここに記録。

### 検討006: DIPの本質的価値 — データカタログ駆動型Retrieval（2026-03-22）

**これはDIPの最も重要な設計決定である。**

#### DIPの本質的価値

DIPは「BigQueryにSQLを投げるチャットボット」ではない。**企業のデータ資産を理解し、適切な権限の下で、適切なデータを選び、専門的な分析を提供するプラットフォーム**である。

#### 実現したい処理フロー

```
① 質問者は誰か（IAP認証）
   │
② Dataplex Data Governance → この人がアクセスできるデータ資産を取得
   │
③ Dataplex Data Catalog → アクセス可能なデータ資産のメタデータを取得
   │  ・テーブルの説明（何のデータか）
   │  ・ビジネス用語（タグ、分類）
   │  ・データ品質スコア
   │  ・データリネージ（由来）
   │
④ テーブル選択AIエージェント → カタログのメタデータを見て関連テーブルを選択
   │  ※ 生のテーブル名・カラム名ではなく、ビジネス的な意味で判断
   │  ※ C07の例外: テーブル選択は「必須処理」ではなく「最適化判断」であり、AIに適した仕事
   │  ※ フォールバック: AIが判断できない場合は全許可テーブルを取得（コード側で保証）
   │
⑤ ルートエージェント → 専門エージェントに振り分け
   │  ・要因分析 / 比較分析 / 予測分析 / 汎用
   │
⑥ 専門エージェント → 選択されたテーブルのみにアクセス＋回答生成
   │
⑦ 回答をユーザーに返す
```

#### 具体例

```
会社全体: 100テーブル
  ↓
Aさんの権限: 20テーブル（Data Governance）
  ↓
質問「安全パトロールの指摘件数は？」
  ↓
AIエージェントがカタログを見て選択: 5テーブル
  ・incident_records（説明: インシデント記録）
  ・quality_inspections（説明: 品質検査結果）
  ・equipment_alerts（説明: 設備アラート）
  ・maintenance_history（説明: メンテナンス履歴）
  ・environment_sensors（説明: 環境センサー）
  ↓
ルートエージェント: 要因分析エージェントに振り分け
  ↓
要因分析エージェント: 5テーブルのみにアクセス→回答生成
```

#### Dataplexが提供するもの

| 機能 | 内容 | DIPでの使い方 |
|------|------|-------------|
| **Data Catalog** | データ資産のメタデータ管理（説明・タグ・分類・ビジネス用語） | AIエージェントがテーブルの「意味」を理解して選択 |
| **Data Governance** | アクセスポリシー、ユーザー別権限管理 | ユーザーが見えるテーブルを制限 |
| **Data Quality** | データ品質ルール（鮮度、完全性、整合性の自動チェック） | 品質が低いデータは回答時に警告 |
| **Data Lineage** | データの由来・変換履歴の追跡 | 出典情報の精度向上 |

#### 関連GCPサービス

| サービス | 内容 | DIPとの関係 |
|---------|------|------------|
| **Analytics Hub** | データセットの共有・公開マーケットプレイス | 複数企業間でのデータ共有（フェーズ3） |
| **Data Lineage API** | 変換パイプラインの追跡 | 「このデータはどこから来たか」を出典に表示 |
| **Policy Tags（BigQuery）** | カラムレベルのアクセス制御 | 機密カラム（給与、個人情報等）のマスキング |
| **Cloud DLP** | 機密データの検出・マスキング | 回答に含まれる個人情報等を自動マスキング |

#### 現状との差分

| ステップ | 現状（v5.9.1） | 目標 |
|---------|--------------|------|
| ① 認証 | ユーザー識別あり（IAP）。テーブル権限なし | ユーザーごとにアクセス可能テーブルをDataplexで定義 |
| ② 権限取得 | なし（全ユーザーが全テーブルにアクセス） | Dataplex Data Governanceから取得 |
| ③ カタログ参照 | なし（テーブル名・カラム名を直接見ている） | Dataplex Data Catalogのメタデータを参照 |
| ④ テーブル選択 | 全テーブルSELECT *（絞り込みなし） | AIエージェントがカタログベースで選択 |
| ⑤ ルーティング | 済み（ADKルーター） | 済み |
| ⑥ データ取得 | 全テーブルのデータを一括注入 | 選択テーブルのみ取得 |

#### 実装ステップ（段階的）

| 段階 | 内容 | 方法 | 時期 |
|------|------|------|------|
| **1（今）** | カタログインターフェース設計＋ローカルJSON版で実装 | config/data_catalog.json + orchestration/data_catalog.py | 今日 |
| **2（近い将来）** | Dataplex Data Catalog APIに接続 | data_catalog.pyの中身をAPI呼び出しに差し替え | 次のセッション |
| **3（PoC後）** | Data Governance連携（ユーザー別権限） | IAP → Dataplex → 許可テーブル取得 | フェーズ3 |
| **4（本番）** | Data Quality連携、Policy Tags、Lineage | 品質スコア表示、カラムマスキング、出典にリネージ | フェーズ3 |

#### 設計パターン

Model Garden・エージェントJSON外部化と同じパターン:
- **インターフェースを先に作り、後からDataplexに差し替え可能にする**
- 段階1: ローカルJSON（Dataplexの代替）
- 段階2以降: Dataplex APIに差し替え。フロントエンド・エンジンのコードは変更不要

```
config/data_catalog.json（ローカル版 — Dataplexの代替）
  ├ テーブルごとのメタデータ（説明、ビジネス用語、タグ）
  ├ アクセス権限マッピング（ユーザー→テーブル）
  └ データ品質スコア

orchestration/data_catalog.py（インターフェース）
  ├ get_accessible_tables(user) → 権限フィルタ済みテーブル一覧
  ├ get_table_metadata(tables) → カタログ情報
  └ select_relevant_tables(question, metadata) → AIエージェントで絞り込み
```

#### C07（LLMにツール実行判断を委ねない）との関係

テーブル選択はC07の**例外**として明確に定義する:
- C07の対象: 「必ず実行すべき処理」の実行判断 → LLMに委ねない
- テーブル選択: 「最適化判断」（どのテーブルが関連するか） → AIに適した仕事
- **ただし安全策**: AIが判断できない場合は全許可テーブルを取得（コード側で保証）
- **ただし上限**: 選択テーブル数の上限をコード側で設定（コスト・速度制御）

**テーブル選択の2段構え方式（2026-03-23確立）**:

| 段階 | 方式 | 速度 | 精度 | 費用 |
|------|------|------|------|------|
| 1段目 | キーワードマッチング（コード側） | 0秒 | 中（テーブル説明の充実度に依存） | 0円 |
| 2段目 | AI選択（Gemini Flash）※1段目で失敗時のみ | 3〜9秒 | 高（文脈理解・因果推論） | Gemini API 1回 |

トレードオフ:
- キーワードマッチは因果関係や抽象概念を推論できない（例:「なぜ生産性が落ちたのか」→maintenance_historyとの関連を推論できない）
- テーブル説明に同義語・関連語を追加すればキーワードマッチの精度が向上する（コストゼロ）
- **データカタログの充実度がDIPの回答品質に直結する**

**データカタログの重要性（2026-03-23確認）**:
- キーワードマッチ方式の導入で、テーブル説明の充実度がDIPの回答品質と速度に**直結**するようになった
- 説明が貧弱→キーワードマッチ失敗→AI選択フォールバック→9秒遅延
- 説明が充実→キーワードマッチ成功→0秒→速度と品質が両立
- **データカタログはDIPの本質的価値を支える基盤である**

**2段構えの安全設計（2026-03-23確認）**:
- 1段目（キーワード）が失敗しても2段目（AI）がカバーする
- 完全にテーブル選択が失敗することはない（全テーブルフォールバックもある）
- ユーザーへの影響は速度のみ（精度は2段目で担保）

同義語・関連語（必須）:
- テーブル説明に、ユーザーが使いそうな表現を含める
- 例: incident_recordsの説明に「指摘、パトロール、安全、事故、ヒヤリハット」を含める
- 管理画面（/admin → データカタログ）から設定可能

**ステータス**: 段階1完了（キーワードマッチ+AIフォールバックの2段構え）。
