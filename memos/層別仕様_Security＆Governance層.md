## 1. 大方針

**物理的な分離をベースとして、その上に論理的なフィルタを追加する。**

- 論理的なフィルタ（コードでのフィルタリング）はバグ1つで漏れる（#006で実証済み）
- 物理的な分離（GCPプロジェクト分離、データストア分離等）はバグがあってもアクセスできない
- 物理分離を土台に、論理フィルタで利便性を高める順序が正しい

## 2. 脅威と対策の全体像

| 脅威 | 深刻度 | 対策 |
|------|--------|------|
| **企業外へのデータ漏洩** | 致命的 | VPC Service Controls、IAM、Cloud DLP、Cloud KMS（CMEK）、Private Google Access |
| **企業内の権限を超えたデータ漏洩** | 致命的（損害賠償沙汰） | GCPプロジェクト分離、データストア分離、データセット分離、IAP認証、エージェントによるロール制御 |
| **過去Q&A経由のデータ漏洩** | 高 | Vertex AI Searchデータストアをロール別に物理分離 |
| **不正なSQL実行** | 高 | sql_validator.py（DML禁止、SELECT/WITH限定、複文防止） |
| **プロンプトインジェクション** | 中 | HTMLエスケープ、ファイル名サニタイズ、GCSサイズ制限 |

## 3. セキュリティ対策一覧（強度順）

| # | 対策 | 強度 | 分離レベル | Pros（利点） | Cons（欠点・制約） | MVP対応 |
|---|------|------|-----------|-------------|-------------------|---------|
| **1** | **GCPプロジェクト分離** | 最高 | インフラ | BigQuery、GCS、Vertex AI Searchがすべて物理的に別リソース。コードにバグがあっても別プロジェクトには一切アクセス不可 | プロジェクトごとに課金・管理が発生。構成が複雑になる。プロジェクト間のデータ連携に追加設計が必要 | ❌ 複数プロジェクト不可 |
| **2** | **VPC Service Controls** | 最高 | ネットワーク | GCPリソースの周囲に「境界」を設定し、境界外への通信をAPIレベルで遮断。認証情報が漏洩してもVPC外からアクセスできない | 組織アカウントが必須。設定ミスで正規の通信も遮断するリスク。デバッグが困難 | ❌ 組織アカウント必要 |
| **3** | **IAP（Identity-Aware Proxy）** | 高 | 認証 | ログインしていないユーザーはアプリ自体にアクセスできない。Google Workspace認証でゼロトラストを実現 | Cloud Run等のデプロイ基盤が必要。Streamlitでは使えない。Workspace契約が前提 | ❌ Cloud Run必要（Phase 5） |
| **4** | **Vertex AI Searchデータストア分離** | 高 | データストア | ロール・データ分類ごとにQ&A記憶を物理分離。フィルタのバグで漏れるリスクがない | データストア数が増える。検索結果の統合が必要な場面で追加ロジックが要る。データストアごとに課金 | ⚠️ 複数作成可能だが現在は1つ運用 |
| **5** | **BigQueryデータセット分離** | 高 | データ | 公開/制限/機密でデータセットを分離。Authorized Viewsで必要な列だけ公開。テーブル単位の精密な制御 | データセット設計が複雑化。Authorized Viewsの管理コスト。データセット間のJOINに制約 | ⚠️ 設計可能だがIAM制御は組織推奨 |
| **6** | **Private Google Access** | 高 | ネットワーク | GCPリソースへの通信がGoogle内部ネットワーク経由。インターネットを経由しないため盗聴リスク排除 | VPCの設定が必要。ローカル開発環境からのアクセスに制約が出る | ❌ VPC設定必要 |
| **7** | **Cloud DLP** | 中〜高 | コンテンツ | データ中身を自動スキャンし個人情報・機密情報を検出。マスキングやトークン化でAIに渡す前に無害化 | API呼び出しコストが高い。大量データのスキャンに時間がかかる。完全な検出精度ではない | ⚠️ API利用可能だが統合は工数大 |
| **8** | **Dataplex** | 中 | ガバナンス | 全ポリシーを一元管理。カタログ、アクセスポリシー、監査ログ、リネージュを統合管理するコントロールプレーン | 組織レベルの設定が前提。学習コストが高い。単体では制御力がなく他サービスとの連携が必須 | ❌ 組織レベルの設定 |
| **9** | **IAM（ロールベースアクセス制御）** | 中 | 認証 | GCPリソースへのアクセスをロール単位で制御。GCPの基本機能で追加コストなし | 設定ミスのリスク。ロール数が増えると管理が煩雑。テストに複数ユーザーが必要 | ⚠️ 設定可能だがテストユーザー1人 |
| **10** | **エージェントによるロール判定** | 中 | アプリ | ログインユーザーのロールでアクセス先を切り替え。DIPコードで柔軟に制御可能 | アプリレベルの論理制御なのでバグで漏れる可能性。認証基盤（IAP等）がないと偽装可能 | ✅ コード実装可能 |
| **11** | **Vertex AI Search企業フィルタ** | 低〜中 | 論理 | 検索時にcompanyでフィルタ。実装済みで追加コストなし | 論理フィルタなのでコードのバグで漏れるリスクあり（#006で実証済み） | ✅ 実装済み |
| **12** | **Cloud KMS（CMEK）** | 補助 | 暗号化 | 暗号化キーを顧客自身が管理。Google側でもキーなしではデータを読めない | 鍵管理の運用負荷。鍵を紛失するとデータが永久に復号不可。追加コスト | ❌ 組織アカウント推奨 |

### 補足: プライベートクラウドの選択肢

GCPでは**Google Distributed Cloud（GDC）**により、GCPの機能をオンプレミスやエアギャップ環境で動作可能。ただしVertex AI SearchやAgent Builderの対応状況に制約がある。現実的には**VPC Service Controls + Private Google Access**でパブリッククラウド上にプライベートクラウドに近い隔離を実現する方法が有力。

## 4. Vertex AI Search（Q&A記憶）のデータ分離設計

### 現在の保存単位

**企業単位。** ユーザーの権限・ロールは考慮していない。

### 問題: ロールベースアクセス制御導入後のリスク

```
Aさん（経営層）が機密データを含む質問 → 詳細な回答が生成 → Vertex AI Searchに保存
Bさん（一般社員）が似た質問 → 過去事例検索でAさんの回答がヒット → 機密データが漏洩
```

エージェントとDataplexで入口を制御しても、**Vertex AI Searchの過去事例経由でデータが漏れる。**

### 将来の保存・検索単位

| 段階 | 保存単位 | フィルタ条件 |
|---|---|---|
| 現在 | 企業 | `company` |
| 将来（Phase 6以降） | 企業 + ロール + データ分類 | `company` + `role_level` + `data_classification` |

将来の保存イメージ:
```json
{
  "question": "部門別利益率を教えて",
  "answer": "...",
  "company": "mazda",
  "role_level": "executive",
  "data_classification": "confidential",
  "intent": "analysis",
  "timestamp": "2026-03-19T..."
}
```

### 補足・予備知識: 一般的な生成AIとDIPの記憶の違い

| | 一般的な生成AI（Claude、ChatGPT等） | DIP |
|---|---|---|
| 記憶の仕組み | なし（会話ウィンドウ内のみ） | Vertex AI Searchに永続保存 |
| 会話を閉じたら | 全部忘れる | 覚えている |
| 別の人が使ったら | 何も引き継がない | 同じ企業なら全員で共有 |
| 使い続けると | 賢くならない | Q&Aが蓄積されて賢くなる |
| 保存期間 | なし | 無期限（手動削除可能） |

「企業単位で、全メンバーのQ&Aを蓄積し、セマンティック検索で類似事例を自動活用する」仕組みは通常の生成AIにはない。**DIPの大きな差別化ポイント。**

## 5. 3段階アクセス制御モデル

| 段階 | 何を制御するか | 担当 | 分離方式 |
|------|---------------|------|---------|
| **第1段階** | どのデータベースにアクセスできるか | エージェント（権限・ロール判定） | 物理（データセット/プロジェクト分離） |
| **第2段階** | どのデータを受け渡してよいか | Dataplex（データ分類・ポリシー） | 物理+論理（行/列レベル制御） |
| **第3段階** | 過去のQ&Aをどの範囲で共有するか | Vertex AI Search（ロール+データ分類フィルタ） | 物理（データストア分離） |

```
ユーザー
  ↓ IAP認証（ログイン + ロール自動判定）
  ↓
第1段階: エージェントがロールを見てアクセス先を決定
  ↓
第2段階: Dataplexがデータ単位で受け渡しを制御
  ↓
第3段階: Vertex AI Searchがロールに応じた過去事例のみ返す
  ↓
生成AIが回答を作成
```

## 6. IAP（Identity-Aware Proxy）

アプリの前に立つ「門番」。Phase 5（Cloud Run移行）で導入。

### 仕組み

```
ユーザー → IAP（門番） → DIPアプリ
         ここで認証チェック
```

- ユーザーがDIPにアクセスすると、IAPがGoogle Workspaceのログイン画面を表示
- ログイン成功 → IAPがユーザー情報（メールアドレス、所属グループ、ロール）をアプリに渡す
- ログイン失敗・未登録 → アプリに到達する前にブロック

### 今のDIPとの違い

| | 今のDIP（Streamlit） | IAP導入後 |
|---|---|---|
| アクセス制限 | なし。URLを知っていれば誰でも使える | Google Workspaceアカウントがないと使えない |
| ログイン | なし | Googleログインが自動表示 |
| ユーザー判定 | できない | メールアドレス・グループ・ロールが自動取得 |
| ロール切り替え | サイドバーで手動選択（偽装可能） | IAMで定義されたロールが自動適用（偽装不可能） |

## 7. 開発時の機密データ保護

DIPは企業の売上・利益・監査データなど機密性の高いデータを扱う。開発・テスト時に本番データが漏洩しないための対策。

### 7-1. 基本対策

| # | 対策 | 説明 | 誰がやるか |
|---|------|------|-----------|
| **D01** | **本番データを使わない** | 開発・テストでは本番データの代わりにダミーデータ（テストデータ）を使用。本番環境に開発者がアクセスすること自体を禁止 | IT部門が環境を分離 |
| **D02** | **データマスキング** | 本番データをコピーして使う必要がある場合、個人名→「田中太郎」、口座番号→「****1234」のように匿名化・マスキングしてから開発環境に投入 | DBA（データベース管理者）またはCloud DLP |
| **D03** | **NDA（秘密保持契約）** | 外注先と秘密保持契約を締結。違反時の損害賠償条項を含む。法的な最低防衛線 | 法務部門 |
| **D04** | **アクセス権限の最小化（Least Privilege）** | 開発者には必要最小限の権限だけ付与。財務データを扱う機能を作るが、財務データそのものは見えない状態で開発 | IT部門（IAM設定） |
| **D05** | **環境分離（Dev / Staging / Prod）** | 開発環境・検証環境・本番環境を完全に分ける。本番環境には運用担当者しかアクセスできない | IT部門（インフラ） |
| **D06** | **監査ログ** | 誰がいつ何にアクセスしたかを全て記録。不正アクセスがあれば追跡可能 | 自動（Cloud Logging等） |
| **D07** | **セキュリティクリアランス** | 特に機密度の高いデータ（内部通報、監査データ）を扱う場合、開発者にバックグラウンドチェック（身元調査）を実施する企業もある | 人事部門・セキュリティ部門 |

### 7-2. 機密度別の運用基準

| 機密度 | データ例 | 開発時の扱い |
|--------|---------|-------------|
| **一般** | 商品マスタ、公開済み売上 | 本番コピーを開発環境で使用可 |
| **社外秘** | 部門別利益率、顧客リスト | マスキング済みデータで開発。本番アクセスは申請制 |
| **極秘** | 内部通報、監査指摘、M&A情報 | ダミーデータのみ。本番環境は専任者のみアクセス。作業時は2名以上で実施（Four Eyes原則） |
| **法規制対象** | 個人情報、財務報告データ | 法律で取り扱いルールが規定（個人情報保護法、金商法等）。違反は行政処分・刑事罰の対象 |

### 7-3. 外注先に対する追加対策

| # | 対策 | 内容 |
|---|------|------|
| **V01** | **作業場所の制限** | 社内またはセキュリティ認定されたオフィスでのみ作業。カフェ等での作業禁止 |
| **V02** | **端末管理** | 会社支給PCのみ使用。私物PC禁止。USBメモリ等の外部メディア使用禁止 |
| **V03** | **ネットワーク制限** | VPN経由でのみ開発環境にアクセス |
| **V04** | **コードレビュー** | 外注先のコードを社内エンジニアが必ずレビュー。バックドアや情報漏洩コードがないか確認 |
| **V05** | **契約終了時のデータ消去** | プロジェクト終了後、外注先の環境から全データを削除し証明書を提出 |

### 7-4. DIP本番導入時の必須対策

| # | 対策 | 対応するセキュリティ対策 |
|---|------|----------------------|
| 1 | 開発環境と本番環境のGCPプロジェクトを分離 | 対策一覧 #1 |
| 2 | 開発環境にはマスキング済みデータのみ投入 | D02 + Cloud DLP |
| 3 | 本番環境へのアクセスはIAP + IAMで厳格に制御 | 対策一覧 #3, #9 |
| 4 | 全操作の監査ログを記録 | Dataplex + Cloud Logging |
| 5 | 外注先にはNDA + 最小権限 + VPNアクセスのみ | D03, D04, V03 |

## 8. MVP（プロトタイプ）での実現範囲

個人環境の制約: GCPプロジェクト1つ、組織アカウントなし、Cloud Runなし（Streamlit）、テストユーザー1人

| 優先度 | 対策 | MVP実現方法 | 目的 |
|--------|------|------------|------|
| **必須** | #11 企業フィルタ | ✅ 実装済み | 最低限の企業間分離 |
| **推奨** | #10 エージェントによるロール判定 | サイドバーでロール選択（経営層/一般）を設けてデモ。選択ロールに応じてアクセス先を切り替える | ロールベース制御のプロトタイプ |
| **推奨** | #4 データストア分離 | データストアを2つ作成（一般用/経営層用）し、ロール選択に連動 | Q&A記憶の物理分離をデモ |
| **推奨** | #5 データセット分離 | BigQueryにpublic/restrictedの2データセットを作り、ロールで切り替え | データアクセス制御をデモ |
| **参考展示** | #1〜#8 | アーキテクチャ図で「本番ではここにVPC/IAP/DLP/Dataplexが入る」と説明 | エンタープライズ版の設計としてアピール |

### MVPの動作イメージ

```
サイドバー: ロール選択 [一般社員 ▼]
  → BigQuery: dataset_public のみ
  → Vertex AI Search: dip-knowledge-general
  → 機密データには一切アクセスしない

サイドバー: ロール選択 [経営層 ▼]
  → BigQuery: dataset_public + dataset_restricted
  → Vertex AI Search: dip-knowledge-executive
  → 機密データを含む詳細分析が可能
```

本番ではIAP認証でロール自動判定になる。MVPではサイドバー選択で代替してデモする。

### 監査・監視（横断）
- **Dataplex**: 誰がいつ何にアクセスしたかの監査証跡
- **Cloud Monitoring**: レイテンシ、エラー率、異常検知
- **Cloud Logging**: 全操作の構造化ログ
