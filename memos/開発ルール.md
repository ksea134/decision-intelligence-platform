## 1. 開発プロセスルール（最重要）

開発・修正のたびに必ず以下のプロセスを実行すること。**例外なし。**

### 1-1. 開発フロー

```
修正・開発の着手
  ↓
実装計画書の提出 ← ユーザーに提示し承認を得る
  ↓
テスト計画書の提出 ← ユーザーに提示し承認を得る
  ↓
実装
  ↓
テスト計画書に基づきテスト実行
  ↓
テスト結果報告書の提出 ← ユーザーに提示し確認を得る
  ↓
ユーザー承認後にcommit / push
```

### 1-2. 実装計画書（修正前に提出）

修正・開発に着手する前に、以下の形式で実装計画書をユーザーに提示すること。

| 項目 | 内容 |
|------|------|
| **修正内容** | 何をどう変更するか |
| **設計方針** | なぜその方式を選ぶのか |
| **削除するファイル** | 廃止するファイルとその理由 |
| **新規作成するファイル** | ファイル名、内容、形式 |
| **修正するファイル** | ファイル名と変更内容 |
| **修正しないファイル** | 影響範囲外であることの明示 |
| **処理フロー（変更後）** | 変更後のデータの流れ |

### 1-3. テスト計画書（修正前に提出）

修正・開発に着手する前に、以下の形式でテスト計画書をユーザーに提示すること。

| 項目 | 内容 |
|------|------|
| **修正内容** | 何をどう変更するか |
| **影響範囲** | 変更が影響するファイル・機能 |
| **テスト項目** | 本ファイルのセクション7から該当するテスト項目を選定。不足があれば追加 |
| **テスト手順** | 各テスト項目の具体的な実行手順 |
| **合格基準** | 各テスト項目の合格条件 |

### 1-4. テスト結果報告書（修正後に提出）

テスト完了後、以下の形式でテスト結果報告書をユーザーに提示すること。

| 項目 | 内容 |
|------|------|
| **テスト日時** | 実施日時 |
| **テスト項目** | 計画書のテスト項目に対応 |
| **結果（OK/NG）** | 各項目の合否 |
| **エビデンス** | 実行コマンド・出力結果・画面確認の記録 |
| **未解決事項** | NGがあれば原因と対策 |

### 1-5. 不明な入力への対応

ユーザーの入力が不明・不完全・タイプミスと思われる場合、推測して行動せず、必ず聞き返すこと。

### 1-6. テスト実施の原則

| # | ルール | 理由（出典） |
|---|--------|-------------|
| P01 | **1回の動作確認で「直った」と判断しない。** 同じ操作を最低3回繰り返す | #007: instruction累積バグは1回目は正常、2回目以降で発覚 |
| P02 | **企業切替テストを必ず実施する。** A→B→Aの順で切り替えて確認 | #007: 企業コンテキストが前の企業のまま残るバグ |
| P03 | **修正前の既存機能が壊れていないことを確認する（回帰テスト）** | 修正の副作用で別機能が壊れるリスク |
| P04 | **テスト計画書にないテスト項目が必要と判断した場合、開発ルール.mdに追加する** | テスト項目は常に最新化する |

## 2. 設計ルール（コーディング）

| # | ルール | 理由（出典） |
|---|--------|-------------|
| C01 | チャット本体（reasoning_engine.py, chat.py）は安易に変更しない | v2.3.0-stableがベース。変更は影響範囲が大きい |
| C02 | 新しいファイルでloggerを使う場合、`import logging` + `logger = logging.getLogger(__name__)`をセットで記述する | #004: logger未定義でStreamlit Cloudがクラッシュした |
| C03 | LLMの不確実性はすべてコード側で吸収する。プロンプト指示だけに頼らない | #001, #002: プロンプト指示だけではGeminiが従わないケースがあった |
| C04 | `validate_sql()`をSQL実行の唯一の入口として統一する | #001: 日本語カラム名の自動バッククォート処理を確実に適用するため |
| C05 | グローバルオブジェクトの状態を直接変更しない。変更する場合は必ずリセット処理を入れる | #007: エージェントのinstructionが呼び出しごとに累積し、企業間データ混入と空応答を引き起こした |

## 3. Gemini API ルール

| # | ルール | 理由（出典） |
|---|--------|-------------|
| G01 | function callingは使わない（v4で廃止済み） | v4再設計で確立 |
| G02 | whileループでGemini APIを呼び出さない | #003: ループは構造的に無限実行のリスクがある |
| G03 | Gemini呼び出し回数は固定値で設計する（「最大N回」ではなく「必ずN回」） | #003: 不定回数は速度・安定性が予測不能 |
| G04 | ストリーミングはテキスト回答生成（最後のCall）にのみ使用する | #003: ストリーミング+function callは不安定 |
| G05 | `generate_content_stream` + `mode="ANY"` の組み合わせは絶対に使わない | #003: 応答遅延・SQL無限繰り返しの直接原因 |
| G06 | `mode="ANY"`を使う場合は必ず`generate_content`（非ストリーミング）で使う | #003: 非ストリームなら安定 |
| G07 | APIパラメータ（`ToolConfig.mode`等）で動作を制御し、プロンプト指示に頼らない | #002: プロンプトで「必ずツールを使え」と書いてもGeminiは無視することがある |

## 5. データソース受け渡しルール

| # | ルール | 理由（出典） |
|---|--------|-------------|
| D01 | 新しいエンジンやエージェントを追加する際は、全データソース（BQ、GCS、ローカル）が正しく渡されているか必ず確認する | #005: ADK移行時にGCSデータの受け渡しが漏れ、報告書が無視された |
| D02 | GCSデータは「接続成功＋中身が空」のケースがある。content lengthの確認が必要 | #005: is_connected=Trueでもcontent=空の企業がある |
| D03 | Q&A保存時と検索時の企業名は同じ値（表示名）を使う。フォルダ名と表示名を混同しない | #006: 企業名の不一致でフィルタが効かなくなるリスク |
| D04 | Vertex AI Searchの企業フィルタは`company: ANY("企業名")`で実装。全角文字を含む企業名でも正常動作を確認済み | 調査結果（2026-03-19） |

## 6. バージョン管理ルール

```
v{Phase}.{Feature}.{Fix}-{サフィックス}
```

| 桁 | 意味 | カウント方法 |
|---|------|------------|
| Phase | 現在のPhase番号 | 開発ロードマップのPhase番号 |
| Feature | Phase内で追加した機能の数 | 機能追加ごとにカウントアップ |
| Fix | 修正した回数 | バグ修正ごとにカウントアップ |

例:
- `v4.7.4` → Phase 4、7機能追加、4回修正
- Phase 5に進んだら → `v5.0.0` からスタート

- 機能追加時・バグ修正時はDIPアプリのバージョン（`ui/components/chat.py`の`__version__`）を必ず更新すること
- バージョン詳細: → `memos/バージョン管理.md`

## 7. テスト項目

大きなアーキテクチャ変更（エンジン追加、データフロー変更、Phase移行等）を行った後は、以下のテストを必ず実施すること。通常の修正でも、影響範囲に応じて該当するテスト項目を選定・実行すること。

### 7-1. データソース受け渡しテスト

| # | テスト項目 | 確認方法 |
|---|-----------|---------|
| T01 | BigQueryデータが回答に反映されるか | 数値を含む質問（「売上教えて」）→ 回答に実数値が含まれること |
| T02 | GCS資料（報告書）が回答に反映されるか | 報告書のある企業で質問 → 回答に報告書固有の内容が含まれること |
| T03 | ローカルファイルが正しく読み込まれるか | knowledge/prompts等の内容が回答スタイルに反映されていること |
| T04 | GCSデータがない企業でもエラーなく動作するか | GCS未設定の企業で質問 → 正常に回答されること |

### 7-2. 企業間データ分離テスト

| # | テスト項目 | 確認方法 |
|---|-----------|---------|
| T05 | 企業Aの回答に企業Bのデータが混入しないか | 企業Aで質問 → 回答に企業B固有のキーワードが含まれないこと |
| T06 | Vertex AI Searchの過去事例が企業間でまたがらないか | 企業Aで検索 → 返却されるQ&Aのcompanyがすべて企業Aであること |
| T07 | 企業切替後に前の企業のデータが残らないか | 企業A→Bに切替 → Bの回答にAのデータが含まれないこと |

### 7-3. 繰り返し・累積テスト

| # | テスト項目 | 確認方法 |
|---|-----------|---------|
| T10 | 同じ企業で3回以上連続質問しても回答が正常か | 同一企業で3回質問 → 3回目も正常な回答が返ること（instruction累積がないこと） |
| T11 | 企業A→B→Aと切り替えた後、企業Aの回答が正常か | 企業切替を往復 → 最後の企業Aの回答にBのデータが含まれないこと |

### 7-4. 汚染データチェック

アーキテクチャ変更後、以下のスクリプトで全企業の過去事例に他企業のデータが混入していないか確認する。

```python
from infra.vertex_ai_search import create_search_client
client = create_search_client(project_id='decision-support-ai')

companies = ['マツダ', 'セブン＆アイ 監査', '製造業デモ｜工場・生産現場',
             '機能検証01｜数値', '機能検証02｜数値＋報告書', '機能検証03｜数値＋多言語']

for c in companies:
    results = client.search(query='売上', company=c, top_k=10)
    mixed = [r for r in results if r['company'] != c]
    status = 'NG — 混入あり!' if mixed else 'OK'
    print(f'{c}: {len(results)}件中 混入={len(mixed)}件 [{status}]')
    for r in mixed:
        print(f'  !! 混入企業: {r["company"]}')
```

### 7-5. Streamlit Cloudテスト

| # | テスト項目 | 確認方法 |
|---|-----------|---------|
| T08 | push後にStreamlit Cloudでエラーなく起動するか | アプリURLにアクセスし企業選択まで確認 |
| T09 | google-adk未インストール環境でフォールバックするか | Streamlit Cloudで質問 → エラーなく回答されること |

## 8. デプロイ・運用ルール

| # | ルール | 理由 |
|---|--------|------|
| O01 | Streamlit起動前に既存プロセスを`pkill -f streamlit`で停止する | 複数プロセス同時動作で応答が極端に遅くなる |
| O02 | Streamlit Cloudへのpush後、必ずアプリの起動確認を行う | #004: ローカルでは動くがCloudではクラッシュするケースがある |
| O03 | 汚染が疑われる場合の削除手順: `vertex_ai_search.py`の`_doc_client.delete_document`で個別削除可能 | #006: 汚染データの緊急対応手順 |
