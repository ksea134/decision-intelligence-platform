## 1. 目的

DIPを本番運用する際、以下を継続的に把握・管理する必要がある。

- **AIの回答は正確か** — ハルシネーション（でっち上げ）が起きていないか
- **速度は許容範囲か** — ユーザーが待たされすぎていないか
- **コストは適正か** — APIの使いすぎで想定外の課金が発生していないか
- **異常は起きていないか** — エラー率の急上昇、サービスダウンの早期検知

これらが見えない状態で運用すると、問題が起きても気づけない。

## 5. 回答品質評価（Vertex AI Evaluation）

### 評価軸

| 軸 | 説明 | 評価方法 |
|---|------|---------|
| **正確性（Accuracy）** | 回答がデータに基づいているか。数値の間違いはないか | BigQueryの元データと回答内の数値を自動照合 |
| **完全性（Completeness）** | 質問に対して十分な情報を提供しているか | GCS資料の活用度、回答の構成要素をチェック |
| **ハルシネーション** | データに存在しない情報をでっち上げていないか | 回答内の主張とデータソースの突合 |
| **一貫性（Consistency）** | 同じ質問に対して矛盾する回答をしていないか | 過去の類似Q&Aとの比較 |
| **有用性（Usefulness）** | 意思決定に役立つ回答になっているか | ユーザーフィードバック（Human in the Loop） |

### 補足・予備知識: なぜAIの品質評価が必要か

一般的なシステムはテスト時に動作が確定する。しかしAIは確率的に回答を生成するため、同じ入力でも出力が変わる。本番環境で継続的に品質を監視しないと、いつの間にか品質が劣化していても気づけない。これは従来のソフトウェアテストとは異なるAI特有の課題。

| コスト項目 | 課金単位 | 削減策 |

| BigQuery | スキャン量 | SELECT * の結果キャッシュ（実装済み） |
| GCS | ストレージ量 + 読み取り回数 | キャッシュ（実装済み） |
| Cloud Run（将来） | リクエスト数 + CPU時間 | オートスケーリングの適切な設定 |

| 対策 | 効果 |
|------|------|

## 7. Human in the Loop

### 目的
AIの回答に対するユーザーフィードバックを収集し、回答品質の改善に活かす。

### フィードバック収集方法（案）

| 方法 | 説明 | Pros | Cons |
|------|------|------|------|
| 👍👎ボタン | 回答の下に良い/悪いの2択 | 手軽。回答率が高い | 理由がわからない |
| 5段階評価 | 回答の下にスター評価 | 定量化しやすい | ユーザーが面倒に感じる |
| コメント入力 | 自由記述で改善点を入力 | 具体的なフィードバック | 入力率が低い |
| 推奨: **👍👎 + 任意コメント** | 2択で手軽に、詳しく書きたい人はコメントも | 手軽さと詳細さのバランス | — |

### 実装済み仕様（v5.8.0）

- 回答の下に👍👎アイコン（Material Symbols）を表示
- 👍押下: 即座に記録、アイコンがグリーンに変化
- 👎押下: コメント入力欄が表示、送信で記録、アイコンがレッドに変化
- 一度評価したら変更不可（同じ回答に対して）

### フィードバックの記録内容

| 項目 | 内容 |
|------|------|
| rating | good / bad |
| question | ユーザーの質問文 |
| answer_preview | 回答の先頭200文字 |
| company | 選択中の企業名 |
| comment | ユーザーのコメント（👎時のみ、任意） |
| engine | v1 / adk |

### フィードバックの確認方法

| 環境 | 確認方法 |
|------|---------|
| **ローカル** | バックエンドのターミナルログで `[Feedback]` を検索 |
| **Cloud Run** | GCPコンソール → Cloud Logging → フィルタ: `[Feedback]` で検索 |
| **将来（#82完了後）** | BigQueryシンク → SQLで集計・分析（`SELECT rating, COUNT(*) FROM dip_ops.feedback GROUP BY rating`） |

### フィードバックデータの活用

- 👎が多い質問パターンの特定 → プロンプト改善
- 企業別・エージェント別の満足度分析 → モデル選択の最適化
- Vertex AI Evaluationの評価基準の校正

### BigQueryシンク（v5.9.1で設定）

Cloud LoggingのDIPログを自動的にBigQueryに転送。

| 設定 | 内容 |
|------|------|
| データセット | `decision-support-ai.dip_ops` |
| シンク名 | `dip-bigquery-sink` |
| フィルタ | Cloud Run `dip` サービスのログのみ |
| 自動生成テーブル | `run_googleapis_com_stdout_YYYYMMDD`（日別） |

### BigQueryビュー（整形済み）

| ビュー名 | 内容 | 主なカラム |
|---------|------|-----------|
| `v_feedback` | 👍👎フィードバック | timestamp, rating, company, comment, question |
| `v_request_logs` | リクエストログ | timestamp, question, company, engine, elapsed_seconds |
| `v_errors` | エラーログ | timestamp, severity, error_message |

### Looker Studio ダッシュボード

BigQueryの `v_feedback` ビューに接続したLooker Studioレポートを作成済み。
- 👍👎比率（円グラフ）
- フィードバック一覧（表）
- URL: Looker Studio（https://lookerstudio.google.com）から閲覧

## 8. リクエストログ（監査証跡）

> 2026-03-20追加: 本番運用で「誰が・いつ・何を質問し・何が返ったか」を追跡するための基盤。

### 記録項目

| 項目 | 説明 | GCPサービス |
|------|------|------------|
| タイムスタンプ | リクエスト受信時刻 | Cloud Logging（自動） |
| ユーザーID | 認証済みユーザー（IAP経由） | Cloud Identity |
| 企業 | 選択中の企業名 | アプリケーションログ |
| 入力元 | スマートカード / チャット入力 | アプリケーションログ |
| 質問文 | ユーザーの入力全文 | Cloud Logging → BigQuery |
| 回答文（先頭500文字） | 長文は切り詰め | Cloud Logging → BigQuery |
| 出典情報 | BQ:/GCS:/LOCAL:のファイル一覧 | アプリケーションログ |
| 使用エンジン | V1 / ADK | アプリケーションログ |
| 応答時間 | 全体 + ステップ別（秒） | Cloud Monitoring カスタムメトリクス |
| ステータス | 成功 / エラー / 空応答 | Cloud Logging |

### 保存先

- **リアルタイム**: Cloud Logging（構造化ログ）
- **永続化・分析**: Cloud Logging → BigQueryシンク（`dip_ops.request_logs`テーブル）
- **検索**: BigQuery SQLで自由にクエリ可能

## 9. プロセストレース（障害切り分け）

> 2026-03-20追加: 1回の質問に対する回答プロセスの各ステップを追跡し、障害発生時に「どこで止まったか」を即座に特定する。

### トレース構造（Cloud Trace）

```
[Span 1] データ読込        0.5秒  ✅ BQ: 12テーブル, GCS: 14ファイル
[Span 2] 過去事例検索      1.2秒  ✅ Vertex AI Search: 3件ヒット
[Span 3] エンジン選択      0.0秒  ✅ ADK → general_agent
[Span 4] BQデータ取得      3.8秒  ✅ query_bigquery: 542行
[Span 5] 回答生成(Gemini)  12.1秒 ✅ 2557文字, model=gemini-2.5-flash
[Span 6] FILESタグ抽出     0.0秒  ✅ 15件
[Span 7] 補足フェーズ      4.2秒  ✅ 思考ロジック+インフォグラフィック
[Total]                    21.8秒 ✅
```

### 実装方法

| 要素 | GCPサービス | 説明 |
|------|------------|------|
| トレース送信 | Cloud Trace | OpenTelemetry SDKでSpanを生成・送信 |
| Span可視化 | Cloud Trace UI | ウォーターフォール表示で各ステップの所要時間を可視化 |
| ログとの紐付け | trace_id | Cloud LoggingのログエントリにtraceIDを付与し、トレースとログを横断検索 |

## 10. 外部モニタリング＆アラート通知

> 2026-03-20追加: DIP本体とは独立した監視基盤から、ログ・メトリクスを監視し、異常時に自動通知する。

### 構成図（全てGCP標準機能）

```
DIP（Cloud Run）
  │
  ├─→ Cloud Logging（構造化ログ送信）
  │     │
  │     ├─→ BigQuery シンク（ログ永続化・SQL分析用）
  │     └─→ ログベースメトリクス（エラー率等をメトリクスに変換）
  │
  ├─→ Cloud Trace（分散トレース送信）
  │     └─→ Span可視化（各ステップの所要時間）
  │
  └─→ Cloud Monitoring（カスタムメトリクス送信）
        │
        ├─→ ダッシュボード
        │     ├─ お客様IT部門が閲覧
        │     └─ Kyndryl運用チームが遠隔閲覧
        │
        └─→ アラートポリシー
              ├─→ お客様IT部門（Cloud Alerting → メール / Slack）
              └─→ Kyndryl運用チーム（Cloud Alerting → PagerDuty / メール）
                    └─→ Kyndryl Bridge Connector（統合運用基盤に集約）
```

### アラートポリシー

| 異常 | 条件 | 重大度 | 通知先 | GCPサービス |
|------|------|--------|--------|------------|
| エラー率超過 | 5分間のエラー率 > 5% | 緊急 | お客様IT + Kyndryl | Cloud Alerting |
| 応答時間SLA超過 | P95 > 30秒（5分間） | 警告 | お客様IT + Kyndryl | Cloud Alerting |
| 空応答多発 | 空応答 > 3回/1時間 | 警告 | Kyndryl | Cloud Alerting |
| API障害 | Gemini/BQ/GCSのHTTP 5xx | 緊急 | お客様IT + Kyndryl | Cloud Alerting |
| コスト超過 | 日次コスト > 予算の120% | 警告 | お客様IT + Kyndryl | Budget Alert |
| サービスダウン | ヘルスチェック連続失敗 | 緊急 | お客様IT + Kyndryl | Uptime Check |

### なぜDIP外部で監視するか

- DIP自体が障害で停止したら、DIP内蔵のダッシュボードも見えなくなる
- お客様のIT部門は既存の監視ツールで一元管理したい
- Kyndryl側が遠隔で状況を把握する必要がある
- Cloud Monitoring/Loggingは**DIPとは独立したGCPサービス**なのでDIP障害の影響を受けない

## 11. 実装タイミング（事業フェーズとの対応）

DIPの事業展開は3段階: **1.開発・検証（今）→ 2.PoC・実証実験 → 3.本番運用**。
AI Opsの各Phaseはこの事業フェーズに紐づく。

| AI Ops Phase | 事業フェーズ | タイミング | 内容 | GCPサービス | 理由 |
|-------------|-------------|----------|------|------------|------|
| **A** | **2. PoC開始前** | Phase 5と同時 | loggerレベル修正、Cloud Logging連携、リクエストログ記録、プロセストレース基盤 | Cloud Logging, Cloud Trace | お客様に渡す前にログ基盤が動いていないと、障害時に何も調べられない |
| **B** | **2. PoC期間中** | Phase 5完了直後 | パフォーマンス統計（P50/P95/P99）、エラー率カウント、基本アラート、通知先設定 | Cloud Monitoring, Cloud Alerting | PoC中にお客様側で問題が起きたとき、お客様IT部門とKyndrylに即通知が必要 |
| **C** | **3. 本番運用** | 運用安定後 | コスト管理ダッシュボード、BigQueryシンク、Lookerレポート | Billing Export, Looker, BigQuery | 正式契約後は予算管理・月次レポートが必須 |
| **D** | **3. 本番運用** | 中長期 | 回答品質自動評価、ユーザーフィードバック、Kyndryl Bridge統合 | Vertex AI Evaluation, カスタム実装 | 継続的な品質改善と統合運用基盤への接続 |

### Kyndryl Bridgeとの連携

7層構造の「AI Ops & Observability」はKyndryl Bridge Connectorを通じて統合運用基盤と接続する設計。Cloud Monitoringのメトリクスをbridge経由でKyndrylの運用ダッシュボードに集約し、DIPを含むIT環境全体の一元監視を実現する。
