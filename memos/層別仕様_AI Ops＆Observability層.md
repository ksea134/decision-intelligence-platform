## 1. 目的

DIPを本番運用する際、以下を継続的に把握・管理する必要がある。

- **AIの回答は正確か** — ハルシネーション（でっち上げ）が起きていないか
- **速度は許容範囲か** — ユーザーが待たされすぎていないか
- **コストは適正か** — APIの使いすぎで想定外の課金が発生していないか
- **異常は起きていないか** — エラー率の急上昇、サービスダウンの早期検知

これらが見えない状態で運用すると、問題が起きても気づけない。

## 3. 機能一覧

| # | 機能 | 説明 | 技術要素 | Pros（利点） | Cons（欠点・制約） |
|---|------|------|----------|-------------|-------------------|
| **O01** | **Metrics収集** | レイテンシ、エラー率、スループット、同時接続数をリアルタイム収集 | Cloud Monitoring | GCPネイティブ。ダッシュボード標準装備。カスタムメトリクスも定義可 | 初期設定にインストルメンテーション（計測コード埋め込み）が必要 |
| **O02** | **ログ統合** | 全操作の構造化ログを集約。誰がいつ何を質問し、どのエージェントがどう回答したか | Cloud Logging | GCPサービスのログは自動収集。フィルタ・検索が高速 | ログ量が増えるとコスト増。保存期間の設定が必要 |
| **O03** | **分散トレーシング** | 1つのリクエストがどのコンポーネントをどの順序で通過したか追跡 | Cloud Trace | ボトルネックの特定が容易。レイテンシの内訳が可視化される | OpenTelemetry等のSDK組み込みが必要 |
| **O04** | **回答品質評価** | ハルシネーション検出、正確性・完全性の自動評価 | Vertex AI Evaluation | LLMの回答品質を定量化できる。継続的な品質監視が可能 | 評価基準の設計が必要。100%の検出精度ではない |
| **O05** | **コスト監視** | トークン使用量、モデル別API課金の可視化と予測 | Billing Export + Looker | コスト超過の早期発見。モデル選択の最適化判断に使える | Billing Exportの設定、Lookerダッシュボードの構築が必要 |
| **O06** | **アラート設定** | エラー率急上昇、レイテンシ閾値超過、コスト超過時に自動通知 | Cloud Alerting | 問題の早期発見。24時間自動監視 | 閾値設定が不適切だと誤報（アラート疲れ）が発生 |
| **O07** | **Human in the Loop** | AIの回答に対するユーザーフィードバック収集。回答の良し悪しを評価 | カスタム実装 | 実ユーザーの満足度を直接計測。評価データがモデル改善に活用可 | UI設計が必要。ユーザーが評価を面倒に感じる可能性 |

## 5. 回答品質評価（Vertex AI Evaluation）

### 評価軸

| 軸 | 説明 | 評価方法 |
|---|------|---------|
| **正確性（Accuracy）** | 回答がデータに基づいているか。数値の間違いはないか | BigQueryの元データと回答内の数値を自動照合 |
| **完全性（Completeness）** | 質問に対して十分な情報を提供しているか | GCS資料の活用度、回答の構成要素をチェック |
| **ハルシネーション** | データに存在しない情報をでっち上げていないか | 回答内の主張とデータソースの突合 |
| **一貫性（Consistency）** | 同じ質問に対して矛盾する回答をしていないか | 過去の類似Q&Aとの比較 |
| **有用性（Usefulness）** | 意思決定に役立つ回答になっているか | ユーザーフィードバック（Human in the Loop） |

### 補足・予備知識: なぜAIの品質評価が必要か

一般的なシステムはテスト時に動作が確定する。しかしAIは確率的に回答を生成するため、同じ入力でも出力が変わる。本番環境で継続的に品質を監視しないと、いつの間にか品質が劣化していても気づけない。これは従来のソフトウェアテストとは異なるAI特有の課題。

## 6. FinOps（コスト管理）

### コスト構成

| コスト項目 | 課金単位 | 削減策 |
|-----------|---------|--------|
| Gemini Flash API | トークン数 | 汎用質問はFlashで処理（現在実装済み） |
| Gemini Pro API | トークン数 | 分析・比較・予測のみProを使用（現在実装済み） |
| Claude API（将来） | トークン数 | Model Gardenで最適モデルを自動選択 |
| Vertex AI Search | ドキュメント数 + 検索回数 | Q&Aの重複排除、古いQ&Aの定期削除 |
| BigQuery | スキャン量 | SELECT * の結果キャッシュ（実装済み） |
| GCS | ストレージ量 + 読み取り回数 | キャッシュ（実装済み） |
| Cloud Run（将来） | リクエスト数 + CPU時間 | オートスケーリングの適切な設定 |

### DIPの現在のコスト最適化（実装済み）

| 対策 | 効果 |
|------|------|
| Gemini Flash/Pro使い分け | 汎用質問のコストを削減（Proの約1/10） |
| BigQuery/GCSキャッシュ | 同じデータの再取得を防止（5分間キャッシュ） |
| Vertex AI Searchクライアントキャッシュ | 接続の再生成を防止（永続キャッシュ） |

## 7. Human in the Loop

### 目的
AIの回答に対するユーザーフィードバックを収集し、回答品質の改善に活かす。

### フィードバック収集方法（案）

| 方法 | 説明 | Pros | Cons |
|------|------|------|------|
| 👍👎ボタン | 回答の下に良い/悪いの2択 | 手軽。回答率が高い | 理由がわからない |
| 5段階評価 | 回答の下にスター評価 | 定量化しやすい | ユーザーが面倒に感じる |
| コメント入力 | 自由記述で改善点を入力 | 具体的なフィードバック | 入力率が低い |
| 推奨: **👍👎 + 任意コメント** | 2択で手軽に、詳しく書きたい人はコメントも | 手軽さと詳細さのバランス | — |

### フィードバックデータの活用

- 👎が多い質問パターンの特定 → プロンプト改善
- 企業別・エージェント別の満足度分析 → モデル選択の最適化
- Vertex AI Evaluationの評価基準の校正

## 8. MVP（プロトタイプ）での実現範囲

| 優先度 | 機能 | MVP実現方法 | 目的 |
|--------|------|------------|------|
| **実装済み** | コスト最適化 | Flash/Pro使い分け、キャッシュ | すでに動作中 |
| **推奨** | 回答時間の計測 | chat.pyでの`time.time()`計測（実装済み） → ログ出力強化 | パフォーマンスの可視化 |
| **推奨** | Human in the Loop | 👍👎ボタンをチャットUIに追加 | ユーザーフィードバック収集の第一歩 |
| **参考展示** | ダッシュボード | モックアップ画像で「本番ではこう見える」と説明 | エンタープライズ版のアピール |
| **将来** | O01〜O06 | Phase 7（Cloud Monitoring統合）で本格実装 | 本番運用の品質保証 |

### Kyndryl Bridgeとの連携

7層構造の「AI Ops & Observability」はKyndryl Bridge Connectorを通じて統合運用基盤と接続する設計。Cloud Monitoringのメトリクスをbridge経由でKyndrylの運用ダッシュボードに集約し、DIPを含むIT環境全体の一元監視を実現する。
